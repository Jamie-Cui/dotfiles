* dotfiles

Personal Linux and macOS configuration deployed with GNU Stow.  Each directory
under =packages/= is an independently deployable package whose contents mirror
paths below the target home directory.

Reusable Codex skills live in the =skills= Stow package and deploy into
=~/.agents/skills=.  Other source repositories remain outside dotfiles
deployment.

** Requirements

- GNU Stow 2.x
- GNU Make
- Standard POSIX tools such as =find=, =grep=, =sed=, and =sh=

On macOS:

#+begin_src bash
brew install stow
#+end_src

On Fedora:

#+begin_src bash
sudo dnf install stow make
sudo dnf copr enable mguessan/davmail
sudo dnf install davmail java-latest-openjdk-headless isync maildir-utils
#+end_src

** Package layout

Most packages mirror =$HOME= directly:

#+begin_example
packages/
├── skills/.agents/skills/
├── vim/.vimrc
├── tmux/.tmux.conf
├── x11/.Xresources
├── nvim/.config/nvim/
├── kitty/.config/kitty/
├── aerospace/.config/aerospace/
├── hypr/.config/hypr/
├── i3/.config/i3/
├── rofi/.config/rofi/
├── fcitx5/.config/fcitx5/
├── gtk-2.0/.gtkrc-2.0
├── davmail/.config/{davmail,systemd/user}/
├── isync/.config/isyncrc
└── bin/.local/bin/
#+end_example

=rime/= is the only special package.  Its files are deployed into a
profile-specific input-method directory rather than directly below =$HOME=.

** Profiles

The Makefile selects =macos= on Darwin and =linux= on Linux.  Override =PROFILE=
only when testing another platform's package set.

| Profile | Regular packages                                                                                                             | Rime target                |
|---------+------------------------------------------------------------------------------------------------------------------------------+----------------------------|
| =macos=   | =vim kitty aerospace bin skills=                                                                                               | =~/Library/Rime=             |
| =linux=   | =vim tmux nvim kitty bin x11 rofi i3 i3blocks hypr waybar dunst flameshot fcitx5 gtk-2.0 gtk-3.0 gtk-4.0 imsettings davmail isync skills= | =~/.local/share/fcitx5/rime= |

The Linux profile installs both i3 and Hyprland configuration.  Missing either
desktop environment does not affect Stow deployment.  Linux defaults to the
Fcitx5 Rime directory; an IBus host can override it in =local.mk=:

#+begin_src makefile
RIME_TARGET_linux = $(DEPLOY_HOME)/.config/ibus/rime
#+end_src

The Linux =fcitx5= package manages input-method hotkeys, the active Rime and US
keyboard profile, and stable addon UI settings.  Generated layout caches and
Rime user data remain outside the package.  The =gtk-2.0= package complements
the GTK 3 and GTK 4 packages and uses the same generated =FONT_SIZE= value.

Reference-only Clang and CMake configuration lives under =templates/=.  These
files are not Stow packages and are never deployed by the Makefile.

Add an optional deployable package without changing a profile:

#+begin_src bash
make stow EXTRA_PACKAGES="nvim tmux"
#+end_src

Set =RIME=0= to skip Rime for a command.

The Linux =davmail= package manages the loopback-only gateway configuration and
its systemd user unit.  OAuth tokens, logs, and =~/.authinfo= credentials remain
outside the repository.  After stowing, enable the service with:

#+begin_src bash
mkdir -p ~/.local/state/davmail
systemctl --user enable --now davmail.service
#+end_src

The Linux =isync= package configures =mbsync= to mirror Outlook mail from the
local DavMail IMAP listener into =~/.local/share/mail/outlook=.  It reads the
login and DavMail token-encryption key from the existing mode-0600
=~/.authinfo= file; no account or secret is stored in dotfiles.  Run
=mbsync outlook= to synchronize.  Emacs mu4e then indexes and reads only this
local Maildir, while outgoing mail continues through DavMail SMTP.

** Agent skills

Keep reusable skill source under =packages/skills/.agents/skills/=.  The
=skills= package is part of every profile, so the standard lifecycle commands
install and refresh it together with the rest of the selected profile:

#+begin_src bash
make dry-run
make stow
make restow
make unstow
#+end_src

The deployment uses =--no-folding= for regular dotfile packages.  The =skills=
package is stowed separately with folding enabled, so each managed skill
directory becomes a symbolic link that Codex can discover while skills from
other sources can coexist in =~/.agents/skills=.  Run the profile's =restow=
target after adding or removing a skill; edits inside an existing skill are
visible immediately.

This repository is the source of truth for the managed skills.  Do not edit the
installed files below =~/.agents/skills=; edit the matching source directory in
=packages/skills/.agents/skills/= instead.  This repository intentionally does
not provide an =npx skills add= package, generated bundle branch, upstream audit,
or separate installer.  GNU Stow is the supported deployment mechanism.

*** Maintaining a skill

Each skill lives in a lowercase, hyphenated directory and must contain a
=SKILL.md= with YAML frontmatter defining =name= and =description=.  Keep helper
scripts, references, assets, evals, licenses, and =agents/openai.yaml= inside the
owning skill directory.

After adding or removing a skill, preview and refresh the selected profile:

#+begin_src bash
make dry-run
make restow
#+end_src

For content-only edits, the existing Stow link exposes the change immediately.
Still inspect the rendered Markdown and run the profile verification before
committing changes that affect the package layout or deployment behavior.

** Usage

Inspect the selected profile and preview all link operations first:

#+begin_src bash
make help
make list-profile
make dry-run
#+end_src

A bare =make= remains a generation-only operation for compatibility with the
existing =~/opt= bootstrap; it never changes =$HOME=.

Deploy, update, or remove a profile:

#+begin_src bash
make stow
make restow
make unstow
#+end_src

All operations use Stow's =--no-folding= mode.  Target directories such as
=~/.config/kitty= remain real directories, while the managed files inside them
are symbolic links.  This keeps application-created runtime state out of the
repository.

Do not use =stow --adopt= during migration: it moves conflicting target files
into the package tree.  Compare and reconcile existing files explicitly.

** Generated configs

Files ending in =.in= are tracked sources.  =make generate= replaces
=@FONT_SIZE@= and writes ignored output files beside the templates:

- =packages/gtk-2.0/.gtkrc-2.0.in= -> =.gtkrc-2.0=
- =packages/gtk-3.0/.config/gtk-3.0/settings.ini.in= -> =settings.ini=
- =packages/gtk-4.0/.config/gtk-4.0/settings.ini.in= -> =settings.ini=
- =packages/hypr/.config/hypr/hyprland.conf.in= -> =hyprland.conf=
- =packages/i3/.config/i3/config.in= -> =config=
- =packages/kitty/.config/kitty/kitty.conf.in= -> =kitty.conf=

The font size defaults to 10 in the Makefile.  For a persistent, machine-local
override, copy =local.mk.example= to the ignored =local.mk= and edit the value:

#+begin_src bash
cp local.mk.example local.mk
#+end_src

For a one-off override, pass =FONT_SIZE= on the command line.  Command-line
values take precedence over =local.mk=:

#+begin_src bash
make generate FONT_SIZE=12
#+end_src

Template files are excluded from Stow.  Generated outputs must exist before
deployment, so =stow=, =restow=, =dry-run=, and =verify= run generation first.
Run =make unstow= before =make clean= to avoid dangling links.

** Verification

The verification target creates a temporary home directory, deploys and
restows the selected profile, checks representative links and template
exclusion—including the =project-plan= skill link—then unstows it and checks
that no links remain:

#+begin_src bash
make verify PROFILE=macos
make verify PROFILE=linux
#+end_src

After editing a tool configuration, also run the tool-specific check where
available:

#+begin_src bash
bash -n packages/rofi/.config/rofi/rofi-drun.sh
bash -n packages/dunst/.config/dunst/reload-and-test.sh
i3 -C -c packages/i3/.config/i3/config
nvim --headless "+checkhealth" +qa
#+end_src

For UI-visible changes to AeroSpace, Kitty, Hyprland, i3, Waybar, Dunst, or GTK,
reload the application and verify the result visually.

** First migration from manual links

Stow rejects pre-existing files and symbolic links that it does not own.  First
run =make verify=, then =make dry-run=.  For each reported conflict:

1. Inspect it with =readlink= or =diff=.
2. Remove only an old link that is confirmed to point into this repository.
3. Preserve and reconcile real files rather than overwriting them.
4. Run =make stow= and repeat =make dry-run= to confirm the target is clean.

** Security

Do not commit secrets, proxy credentials, private keys, generated Rime user
data, or application caches.  Keep host-specific values outside this repository
when possible.
